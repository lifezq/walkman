## ✅ 第四阶段：技术配置（预计 2-3 天）

### 代码修改
- [ ] 修改应用包名
  - Android：`android/app/build.gradle.kts` 中的 `applicationId`
  - iOS：Xcode 中的 Bundle Identifier
- [ ] 修改应用显示名称
  - Android：`AndroidManifest.xml` 中的 `android:label`
  - iOS：`Info.plist` 中的 `CFBundleDisplayName`
- [ ] 更新版本号：`pubspec.yaml` 中的 `version: 1.0.0+1`
- [ ] 移除所有调试代码和 print 语句
- [ ] 移除未使用的代码和资源

### 应用签名配置
**Android：**
- [ ] 生成 keystore 文件
  ```bash
  keytool -genkey -v -keystore walkman-release.jks \
    -keyalg RSA -keysize 2048 -validity 10000 \
    -alias walkman
  ```
- [ ] 创建 `android/key.properties` 文件
- [ ] 配置 `android/app/build.gradle.kts` 签名信息
- [ ] 安全保存 keystore 文件和密码（永久有效，丢失无法找回）

**iOS：**
- [ ] 在 Apple Developer 创建 App ID
- [ ] 创建发布证书（Distribution Certificate）
- [ ] 创建描述文件（Provisioning Profile）
- [ ] 在 Xcode 中配置 Signing & Capabilities

### 代码混淆（Android）
- [ ] 创建 `android/app/proguard-rules.pro`
- [ ] 在 `build.gradle.kts` 中启用混淆
- [ ] 测试混淆后的 APK 功能正常

### 权限优化
- [ ] 确认所有权限都是必需的
- [ ] 移除未使用的权限
- [ ] 为每个权限准备使用说明
- [ ] 确保权限申请时机合理

### 首次启动优化
- [ ] 添加隐私政策弹窗（首次启动）
- [ ] 用户同意后才开始使用
- [ ] 权限申请时显示使用说明

## ⚠️ 重要提醒

1. **包名修改优先级最高**
   - 包名一旦发布不可更改
   - 建议使用：`com.yourcompany.walkman` 或 `com.yourdomain.walkman`
   - 避免使用 example 域名

2. **keystore 文件妥善保存**
   - Android 签名 keystore 永久有效
   - 丢失后无法更新应用
   - 建议多地备份（云盘、U盘等）

3. **隐私政策必须有效**
   - 必须是独立的网页链接
   - 内容必须完整详细
   - 这是各市场审核的重点

4. **版本号规范**
   - 首次发布建议使用 1.0.0
   - 后续小更新：1.0.1, 1.0.2
   - 功能更新：1.1.0, 1.2.0
   - 大版本更新：2.0.0

5. **测试要充分**
   - 在多个设备上测试
   - 覆盖不同系统版本
   - 确保核心功能稳定
   - 无明显 bug 和崩溃

6. **审核耐心等待**
   - 不同市场审核周期不同
   - 被拒是正常现象
   - 根据反馈修改后重新提交
   - 保持礼貌的沟通
